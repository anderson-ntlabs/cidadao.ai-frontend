## Overview

The Cidadão.AI frontend implements **Server-Sent Events (SSE)** for real-time streaming of AI agent responses. This provides a smooth, incremental user experience where chat responses appear token-by-token as they're generated by the backend.

## Architecture

### Components Involved

```
User Input → ChatService → PrimaryAdapter → SSE Stream → UI Updates
                              ↓
                         EventSource API
                              ↓
                    Backend /api/v1/chat/stream
```

### Key Files

- `lib/chat/adapters/primary.adapter.ts` - Main SSE client implementation
- `lib/chat/adapters/fallback.adapter.ts` - Non-streaming fallback (Maritaca)
- `lib/chat/chat.service.ts` - Orchestrates adapter selection
- `components/chat/message-content.tsx` - Renders streaming messages

## Implementation Details

### 1. SSE Client (Primary Adapter)

```typescript
// lib/chat/adapters/primary.adapter.ts

async send(request: ChatRequest): Promise<ChatResponse> {
  return new Promise((resolve) => {
    const url = new URL(`${BACKEND_URL}/api/v1/chat/stream`)
    url.searchParams.set('message', request.message)
    if (request.agentId) url.searchParams.set('agent_id', request.agentId)
    if (request.sessionId) url.searchParams.set('session_id', request.sessionId)

    const eventSource = new EventSource(url.toString())
    let accumulatedResponse = ''

    eventSource.onmessage = (event) => {
      const data = JSON.parse(event.data)

      if (data.type === 'token') {
        accumulatedResponse += data.content
        // Incremental UI update via callback
        request.onProgress?.(accumulatedResponse)
      }

      if (data.type === 'done') {
        eventSource.close()
        resolve({
          success: true,
          data: {
            response: accumulatedResponse,
            agentId: data.agent_id,
            sessionId: data.session_id
          }
        })
      }
    }

    eventSource.onerror = () => {
      eventSource.close()
      resolve({
        success: false,
        error: { code: 'SSE_ERROR', message: 'Stream failed' }
      })
    }
  })
}
```

### 2. Message Event Types

The backend sends three types of events:

| Event Type | Description                    | Example                                       |
| ---------- | ------------------------------ | --------------------------------------------- |
| `token`    | Single token from LLM response | `{"type": "token", "content": "O "}`          |
| `metadata` | Context info (agent, session)  | `{"type": "metadata", "agent_id": "abaporu"}` |
| `done`     | Stream completion signal       | `{"type": "done", "session_id": "abc123"}`    |

### 3. Progress Callback Pattern

```typescript
// Usage in chat components
const sendMessage = async (message: string) => {
  setStreamingMessage('') // Clear previous

  await chatService.sendMessage({
    message,
    agentId: selectedAgent,
    onProgress: (partial) => {
      setStreamingMessage(partial) // Update UI incrementally
    },
  })
}
```

### 4. Fallback Strategy

If SSE fails or is unavailable:

```typescript
// Automatic fallback to Maritaca (no streaming)
if (primaryResponse.success === false) {
  return fallbackAdapter.send(request) // Returns complete response
}
```

## Error Handling

### Connection Errors

```typescript
eventSource.onerror = (error) => {
  console.error('SSE connection error:', error)

  // Automatic fallback to non-streaming adapter
  return fallbackAdapter.send(request)
}
```

### Timeout Handling

```typescript
const timeout = setTimeout(() => {
  if (!completed) {
    eventSource.close()
    resolve({ success: false, error: { code: 'TIMEOUT' } })
  }
}, 30000) // 30s timeout
```

### Network Recovery

EventSource has built-in reconnection:

- Automatically reconnects on connection drop
- Uses `Last-Event-ID` for resumption
- Exponential backoff (1s, 2s, 4s...)

## Performance Optimizations

### 1. Token Batching

Instead of updating UI on every single token:

```typescript
let buffer = ''
let lastUpdate = Date.now()

eventSource.onmessage = (event) => {
  buffer += event.data

  // Update UI max once per 50ms
  if (Date.now() - lastUpdate > 50) {
    request.onProgress?.(buffer)
    lastUpdate = Date.now()
  }
}
```

### 2. Connection Pooling

Reuse EventSource connections when possible:

```typescript
const connectionCache = new Map<string, EventSource>()

function getOrCreateConnection(url: string): EventSource {
  if (connectionCache.has(url)) {
    return connectionCache.get(url)!
  }

  const es = new EventSource(url)
  connectionCache.set(url, es)
  return es
}
```

### 3. Memory Management

```typescript
// Always close connections when done
eventSource.close()
connectionCache.delete(url)
```

## Backend Requirements

The backend must:

1. **Support SSE protocol**: Set headers:

   ```
   Content-Type: text/event-stream
   Cache-Control: no-cache
   Connection: keep-alive
   ```

2. **Send properly formatted events**:

   ```
   data: {"type":"token","content":"Hello"}\n\n
   data: {"type":"done","session_id":"abc"}\n\n
   ```

3. **Handle CORS**: Allow EventSource requests from frontend origin

4. **Manage connections**: Clean up on client disconnect

## Migration from WebSocket

SSE was chosen over WebSocket because:

- ✅ Simpler protocol (HTTP/1.1)
- ✅ Built-in reconnection
- ✅ Works through most proxies/firewalls
- ✅ No need for bidirectional communication
- ✅ Better browser support
- ✅ Lower overhead for one-way streaming

## Future Enhancements

1. **Compression**: Use gzip for token streams
2. **Multiplexing**: Multiple concurrent streams
3. **Priority**: Urgent messages bypass queue
4. **Resumption**: Continue interrupted streams
